## Context

`autojs6-dev-tools-plus` 是一个新项目，目标不是在现有 WinUI 3 工程内做兼容层，而是保留既有业务约束与能力边界，在新仓库中以 Avalonia 重建跨平台桌面实现。现有 `autojs6-dev-tools` 已经验证了核心业务模型：ADB 设备管理、截图拉取、UI dump 解析、双核独立架构、双路径代码生成、实时匹配测试，这些能力必须在 Plus 项目中完整对齐。

当前约束如下：
- UI 技术栈由用户明确指定为 Avalonia。
- 首版必须覆盖 Windows/macOS；Linux 不承诺首版完整交付，但架构、依赖、渲染、打包设计必须保留支持路径。
- 必须保留双核独立架构：图像引擎与 UI 图层分析引擎完全解耦。
- 必须保留单向依赖：`App → Infrastructure → Core ← Infrastructure`。
- 必须继续遵守 AutoJS6 生成约束：Rhino 循环体内禁止 const/let、OOM 预防、region 优先、坐标左上角原点、路径统一正斜杠。
- 必须继续使用 AdvancedSharpAdbClient 底层 API，不能退回 adb shell 命令式实现。

## Goals / Non-Goals

**Goals:**
- 在独立项目中建立跨平台桌面基础，保证业务能力与 `autojs6-dev-tools` 对齐。
- 用 Avalonia 重建可交互画布、属性面板、设备管理面板、代码预览面板和主工作台。
- 将 WinUI/Win2D 专属逻辑替换为跨平台实现，但不改变核心业务模型、数据模型和生成规则。
- 建立 Windows/macOS 可运行的首版交付路径，并为 Linux 构建、依赖和后续验证保留可执行设计。
- 抽象文件系统、剪贴板、快捷键、窗口、渲染缓存等平台相关服务，避免 App 层再次被单平台绑定。

**Non-Goals:**
- 不在本提案中要求与旧项目共享同一套 UI 代码。
- 不承诺首版 Linux 全量验收通过，只保留设计与实现路径。
- 不改变 AutoJS6 代码生成语义，不新增混合模式。
- 不实现 iOS 支持、云同步、远程协作、脚本运行时或调试器。
- 不回写或重构现有 `autojs6-dev-tools` 仓库。

## Decisions

### 1. 采用独立仓库而不是在旧仓库内硬做多端兼容

**决策：** `autojs6-dev-tools-plus` 作为独立项目存在，单独维护其 OpenSpec、项目结构和跨平台实现。

**理由：**
- 现有项目 App 层深度绑定 WinUI 3 / Win2D / Windows App SDK，继续在原仓库做兼容层会显著增加分支复杂度。
- 独立仓库可以让旧项目继续承担 Windows 原型验证，不阻塞 Plus 项目的跨平台路线。
- 新仓库更容易在初始阶段把平台抽象、目录结构、依赖边界一次性定正确。

**替代方案：** 在原项目内通过条件编译同时支持 WinUI 和 Avalonia → 被拒绝，因 UI、输入、渲染和打包模型差异过大，复杂度过高。

### 2. App 层采用 Avalonia，自定义双图层画布而非追求 Win2D 等价 API

**决策：** 使用 Avalonia 作为 UI 框架，在 App 层构建自定义画布组件与分层渲染模型，实现 ImageLayer + OverlayLayer 的独立更新。

**理由：**
- Avalonia 原生支持 Windows/macOS/Linux，适合当前桌面工具场景。
- Win2D 是 Windows 独占，不存在真正跨平台的 1:1 替代；正确做法是保留“分层渲染原则”，而不是强求 API 等价。
- 自定义双图层画布可继续满足缩放、平移、旋转、命中检测、裁剪和 overlay 独立刷新需求。

**替代方案：** 依赖平台特定图形库分别实现每个平台渲染 → 被拒绝，因维护成本高，且与单一 App 层目标冲突。

### 3. Core 层必须去平台化并隔离任何 Windows-only 包

**决策：** Core 保持纯 .NET 业务层，只包含接口、模型、规则、坐标、代码生成、选择器生成、解析逻辑；任何外部库绑定放入 Infrastructure。

**理由：**
- Plus 项目不能重复当前项目里 `Core` 引入 Windows 定向 OpenCV 包的做法。
- 去平台化后，跨平台测试可直接覆盖核心规则，降低 UI 重写风险。
- 代码生成、UI dump 解析、业务约束是最稳定的资产，应成为可长期复用的核心。

**替代方案：** 继续在 Core 中直接引用图像/OpenCV 实现 → 被拒绝，因会污染核心边界，破坏跨平台目标。

### 4. Infrastructure 封装跨平台图像与 OpenCV 绑定，并允许后续替换实现

**决策：** Infrastructure 统一封装：ADB、图像读写/裁剪、模板匹配、文件导出、平台服务。OpenCV 绑定选择以跨平台可用性为第一优先，而不是跟随旧项目的 Windows 包名。

**理由：**
- 用户仅明确了 Avalonia，未要求固定 OpenCV 绑定实现；因此设计应强调“接口稳定、实现可替换”。
- Windows/macOS/Linux 的原生库装载方式不同，封装层必须吸收这些差异。
- 这样可以在 Windows/macOS 先跑通，再逐步补齐 Linux 的 native 依赖策略。

**替代方案：** 直接把旧仓库的 OpenCvSharp4.Windows 方案照搬 → 被拒绝，因不满足跨平台。

### 5. 功能对齐以“能力对齐”为准，而不是逐文件或逐控件完全复制

**决策：** “100% 一样”在设计上解释为：现有 OpenSpec 能力与关键工作流必须全部覆盖，允许 UI 细节、控件样式、渲染实现、快捷键底层接线方式存在平台适配差异。

**理由：**
- 跨平台桌面应用天然存在菜单、文件对话框、输入法、窗口行为差异，逐像素复制旧 UI 不现实。
- 用户真正关心的是能力和工作流一致：截图、裁剪、匹配、选择器、代码生成、验证、导出。
- 以 capability parity 作为验收标准，更可测试，也更适合 OpenSpec 描述。

**替代方案：** 追求与 WinUI 完全一致的像素级 UI → 被拒绝，因会拖累跨平台交付且无法形成稳定验收标准。

### 6. 首版平台策略：Windows/macOS 实交付，Linux 保留支持设计与构建检查

**决策：** 首版建立 Windows/macOS 运行、打包和基本验证闭环；Linux 先完成架构兼容、依赖探测、CI 构建与关键能力 smoke check 设计。

**理由：**
- 这是用户明确要求。
- Linux 上 OpenCV native 依赖、桌面环境差异、AOT/打包策略更复杂，放在首版完全验收会显著放大风险。
- 先把 Linux 保留在设计和构建层，可以避免后续返工，同时不阻塞首版价值交付。

**替代方案：** 三平台同等深度首发 → 被拒绝，因交付风险过高。

### 7. 延续双核独立、坐标直对齐、双路径代码生成等核心原则

**决策：** Plus 项目完整继承以下架构原则：
- 图像引擎与 UI 图层分析引擎完全独立
- 图像坐标与 dump 坐标都采用左上角原点
- 图像模式与控件模式代码生成路径独立
- region 优先、单轮单截图、及时回收图像资源
- Layout 容器过滤规则不变

**理由：**
- 这些原则来自业务事实和 AutoJS6 约束，而不是 Windows UI 技术栈特性。
- 若在跨平台迁移时打破这些原则，会导致生成代码与真实工作流偏离。

**替代方案：** 借迁移机会重定义坐标模型或合并双引擎 → 被拒绝，因违背现有项目最高优先级约束。

## Risks / Trade-offs

### [风险] Avalonia 自定义画布在高分辨率截图与大量 overlay 下达不到既有流畅度
**缓解措施：** 从设计上保留分层缓存、局部失效刷新、位图缓存池、阈值滑动只更新匹配层，并在任务中加入 Windows/macOS 基准测试。

### [风险] 跨平台 OpenCV native 依赖在 macOS/Linux 安装和分发上更复杂
**缓解措施：** 将匹配能力放在 Infrastructure 可替换实现下，先完成 Windows/macOS 依赖闭环，再为 Linux 制定单独 native 依赖和 CI 校验方案。

### [风险] 功能对齐口径模糊，导致 later stage 验收争议
**缓解措施：** 用 capability specs 明确功能边界，以现有 Windows 版 OpenSpec 为基线，按 capability parity 验收，而非靠主观“看起来差不多”。

### [风险] 旧项目未完成，直接迁移可能把未验证设计一并复制过去
**缓解措施：** Plus 项目只继承已确认的架构原则和能力边界，不盲目复制 WinUI 层代码结构；对平台相关实现重新设计。

### [权衡] 独立仓库会带来两套项目维护成本
**接受理由：** 这是换取跨平台演进空间和避免旧仓库被条件编译污染的必要代价。

### [权衡] Linux 首版不做完整交付会降低“全平台”即时满足感
**接受理由：** 用户已明确 Windows/macOS 优先；把 Linux 纳入设计与构建路径，比仓促承诺完整交付更稳妥。
